# Kobra S1 Bot Wiki

Бот читает сообщения в Telegram (группы, супергруппы, личные чаты), пытается найти релевантную страницу в вашей вики (по `WIKI_SITEMAP_URL`, обычно это `sitemap.xml` на сайте вики) и, если совпадение достаточно хорошее, отвечает ссылкой на страницу.

Краткая инструкция для участников чата: [`wiki/getting-started.md`](wiki/getting-started.md).

## Структура кода

Логика бота разнесена по пакету `app/bot/` (точка входа — `python -m app.bot`, см. `app/bot/__main__.py` и `app/bot/lifecycle.py`):

| Модуль | Роль |
|--------|------|
| `constants.py` | константы, пути к локальным JSON в `.cache/` |
| `i18n.py` | язык ответа (ru/en), строки интерфейса |
| `stores.py` | кэш ответов, feedback после `/error`, фиксы ссылок `/fix` |
| `text_heuristics.py` | эвристики: модель в тексте, тема, код ошибки |
| `wiki_ranking.py` | поиск по индексу с бонусами/штрафами по URL и модели |
| `error_codes_wiki.py` | выбор страницы `/error-codes/...` |
| `error_display.py` | карточки кодов ошибок и перевод |
| `design_replies.py` | короткие ответы из справочника (ACE слот/чип, отрыв TPU, резонанс/PA, ушко в слайсере, отверстия, дверь — без нерелевантной вики) |
| `clarify.py` | уточнение модели и цепочки reply |
| `layer_model_gate.py` | отсев кандидатов вики по модели принтера |
| `handlers.py` | команды и обработка сообщений |
| `admin_access.py` | кто считается администратором для служебных команд |
| `help_text.py` | текст команды `/help` (plain text, ru/en; отдельно для админа и участника) |
| `ephemeral.py` | автоудаление пары «команда + ответ бота» в группах (не в личке и не в чатах из `ephemeral_exempt_chat_ids`) |
| `review_mention.py` | @ревьюер в конце ответа в группах (`REPLY_REVIEW_MENTION`) |
| `reply_logging.py` | лог исходящих ответов (`bot_reply` для зеркала в Telegram) |
| `decision_log.py` | ссылки на сообщения в чате, текст вопроса для лога |
| `telegram_log_mirror.py` | зеркало в служебный чат: ответы бота, индексация, git (без шума polling/apscheduler) |
| `ops_notify.py` | служебные уведомления: старт, ошибки, перезапуски (`OPS_NOTIFY_CHAT_ID`) |
| `git_autopull.py` | фоновый `git pull` (опционально, см. `GIT_AUTOPULL_*`) |
| `reply_access.py` | проверка `getChatMember` — может ли бот писать в чат/тему |
| `manual_qa.py` | ручные пары вопрос–ответ (`/qaadd`, `data/manual_qa.json`) |
| `lifecycle.py` | логирование, `Application`, индексация, polling |

В корне пакета `app/` также:

- **`config.py`** — загрузка настроек из `.env`
- **`ru_layer.py`** — расширение русских запросов англ. ключевыми словами для англоязычной вики (`RU_LAYER_ENABLED`)
- **`wiki_index.py`**, **`web_wiki_index.py`** — индекс страниц вики
- **`default_developers.py`**, **`default_ephemeral_exempt.py`**, **`default_ops_chat.py`** — дефолты для разработчиков, exempt-чатов и `OPS_NOTIFY_CHAT_ID`

В корне репозитория:

- **`sitemap.xml`** — снимок карты сайта вики (для справки и офлайн-просмотра). Рабочий URL задаётся в `.env` как `WIKI_SITEMAP_URL`
- **`data/manual_qa.json`** — ручные Q&A (команды `/qaadd`, `/qalist`, `/qadel`)
- **`wiki/`** — короткие markdown-инструкции для пользователей

Каталог **`logs/`** — ротируемый файл **`logs/bot.log`** (решения бота, индексация, ошибки). Каталог **`.cache/`** создаётся при работе бота (индекс, состояние, локальные сторы) и **не коммитится** (см. `.gitignore`).

## Что нужно

- Python 3.11+ (желательно)
- Токен Telegram-бота из BotFather

## Быстрый старт

1) Установить зависимости:

```bash
python -m venv .venv
.\.venv\Scripts\activate
pip install -r requirements.txt
```

2) Создать `.env` рядом с `requirements.txt` (можно скопировать из примера):

```bash
copy .env.example .env
notepad .env
```

Важно: файл `.env` **не коммитьте** — он добавлен в `.gitignore`. В репозиторий коммитится только `.env.example`.

Минимально нужно заполнить:

- `TELEGRAM_BOT_TOKEN`
- `WIKI_BASE_URL`
- `WIKI_SITEMAP_URL`

3) Запустить:

```bash
python -m app.bot
```

Или (Windows) открыть отдельное окно с логами:

```bat
start-bot.cmd
```

Или (Linux/macOS):

```bash
./start-bot.sh
```

Остановить (если запускали через `start-bot.cmd`):

```bat
stop-bot.cmd
```

Перезапустить:

```bat
restart-bot.cmd
```

На Linux: `./stop-bot.sh`, `./restart-bot.sh`.

4) Добавить бота в группу и написать в группе команду:

`/id`

Бот ответит числовым `chat_id`. Если нужно ограничить бота только выбранными чатами или темами форума — см. `ALLOWED_CHAT_IDS` и `ALLOWED_TOPIC_IDS` в `.env.example` и команду `/status` (показывает, разрешён ли текущий чат/тема).

## Команды бота

- `/help` — справка **обычным текстом** (без HTML): для **администраторов чата** — полный список служебных команд; для **участников** — как задавать вопросы и что служебные команды им недоступны. В **личке** служебные команды доступны.

Служебные команды ниже в **группах и супергруппах** обрабатываются только если вызывающий — **администратор или владелец** этого чата, либо user_id из списка **разработчиков** (`app/default_developers.py` и `DEVELOPER_USER_IDS`); иначе бот **не отвечает** на команду (молча игнорирует). В личке с ботом служебные команды доступны.

- `/id` — ID чата и тип чата
- `/admincheck` — в группе: строка `get_chat_member` и флаги «разработчик» / «доступ к служебным командам» у бота
- `/wiki <запрос>` — поиск по вики по запросу
- `/ping`, `/status` — диагностика
- `/error` (reply на сообщение бота) — плохой ответ: перепоиск и обучение «не эта ссылка»
- `/fix <url>` (reply на сообщение бота) — указать правильную ссылку для того же запроса
- `/qaadd`, `/qalist`, `/qadel` — ручные пары вопрос–ответ (`data/manual_qa.json`; см. `/help`)
- `/update` — подтянуть код с GitHub и перезапустить бота (только админы чата / личка; нужны права на `git` в каталоге клона)

## Как это работает

- Из `WIKI_SITEMAP_URL` берётся список страниц
- Страницы скачиваются и складываются в кэш (`CACHE_PATH`, см. `.env.example`); индексация идёт пакетами (`INDEX_BATCH_SIZE`, `INDEX_INTERVAL_SECONDS`)
- Для каждой страницы строится «текст документа» (title + текст страницы без скриптов/стилей)
- Для русских вопросов при `RU_LAYER_ENABLED=true` запрос расширяется англ. ключевыми словами (`app/ru_layer.py`) — так проще матчить англоязычную вики
- Для каждого сообщения считается похожесть (RapidFuzz `token_set_ratio`), плюс эвристики по модели, URL и кодам ошибок
- Сначала проверяются ручные Q&A, справочник конструкции и каталог кодов ошибок; затем поиск по индексу вики
- Если лучший результат ≥ `MIN_SCORE`, бот отвечает ссылкой (с опциональными уточнениями модели и защитой от нерелевантных URL)
- Если score между `CLARIFY_MIN_SCORE` и `MIN_SCORE`, бот может задать уточняющий вопрос (`CLARIFY_ENABLED`)
- В группах в конце ответа может добавляться @ревьюер (`REPLY_REVIEW_MENTION`) — чтобы приходило уведомление

## Настройки

Смотрите `.env.example`.

Полезное:

- `MIN_SCORE`: порог (0..100). Если слишком много ложных — поднимите.
- `QUESTIONS_ONLY=true`: отвечать только на вопросы («?» или вопросительные слова).
- `REQUIRE_TRIGGER`: в группах для **не-вопросов** требовать @бота или reply на бота; на вопросы (`QUESTIONS_ONLY`) бот отвечает без @.
- `RU_LAYER_ENABLED`: русский слой для поиска по англоязычной вики.
- `CLARIFY_*`: уточняющие вопросы при «почти уверенном» score.
- `REPLY_REVIEW_MENTION`: username (без @) для упоминания в конце ответа в группах; `off` — выкл.
- `ALLOWED_CHAT_IDS` / `ALLOWED_TOPIC_IDS`: ограничить работу конкретными группами и темами.
- `REQUIRE_CAN_REPLY`: перед обработкой проверять `getChatMember` (бот может писать в чат/тему).
- `DEVELOPER_USER_IDS`: дополнительные user_id разработчиков (к `app/default_developers.py`): в группах — служебные команды как у админа, без антиспама и кулдаунов clarify.
- `EPHEMERAL_EXEMPT_CHAT_IDS`: чаты, где не автоудалять пару «команда + ответ» (к `app/default_ephemeral_exempt.py`).
- `OPS_NOTIFY_CHAT_ID`, `OPS_LOG_MIRROR_*`: служебный чат и зеркало лога (см. ниже).
- `WIKI_MAX_PAGES`: ограничить число страниц в индексе (быстрее первый запуск).
- `MEMORY_LIMIT_MB` (Linux/macOS): потолок виртуальной памяти (`RLIMIT_AS`); `0` — без лимита.
- `LOG_DECISIONS`: в `logs/bot.log` строки `seen chat=` и `skip … reason=` — почему бот промолчал.
- Бытовой чат (`text_heuristics.py`, `skip reason=conversational_chatter`): сравнения принтеров, обсуждение покупки б/у, планы «попробую xyz-куб…», сарказм в треде, статусы печати и т.п. — без ссылки из вики.
- Подача филамента / шестерня без модели в тексте — уточнение принтера; не отдаётся общий `extruder-replacement` чужой линейки (например Kobra Max на «шестерни не тянут?»).
- Обсуждение прочности материала («мне интересно… TPU… послойно / поперёк») — бытовой чат; слово «послойно» не путается с настройкой высоты слоя.
- Любопытство про чужие принты («как они так печатают… не похоже на 3D / на видео кажется») — бытовой чат, без ссылки на вики.
- Короткие обрывки сравнения («как кобра х», «как на кобре» без «?») — не ссылка на обзор принтера в вики.
- Пересказ из другого чата («в чате по чиди увидел инфу, что…») — бытовой чат, без clarify и без resin/стола в вики.
- Сарказм в треде («говорит многоцвет не печатает, зачем ему две аськи?») — не clarify и не multi-color вики.
- Комментарии про чужой многоцвет («наигрался с быстрым многоцветом», пресет Slicemaker) — бытовой чат, не Multi-color Printing Guide.
- Личные истории про другие принтеры (Bambu/P2S, «нажрался — потом разобрал экструдер») — бытовой чат, не гайд Kobra по экструдеру.
- Мнение про слайсер («зачем для кобры orca, стандартный слайсер огонь») — бытовой чат, не quick start Orca.
- Вопросы про резонанс/PA/затухающие колебания без модели в тексте — текстовый ответ (не clarify и не layer-shift чужой линейки); «пока слой печатает, есть ещё вопрос» не считается только статусом.
- Наблюдения про ChiTu Box / ACE («на катушку движок», «как я понял, в аське 2 тоже») — бытовой чат, не гайд по замене филамента.
- Вопросы по UI слайсера («как убрать ушко», не получается) — текстовый ответ про mouse ear/brim, не quick start вики; «подскажите» распознаётся как запрос помощи.
- «Какие советы оторвать TPU с пластины» без модели — текстовый ответ про отрыв, не `print-tpu` Kobra 3 Combo; «какие» в общих советах не путается с выбором марки пластика.
- ACE Pro: слот запомнил PETG по чипу, «сбросить настройки» — ответ про слот/RFID без уточнения модели принтера; слово «настройки» не путается с калибровкой стола.
- Мнение о качестве стороннего пластика (Eryone и т.п., «сломал на столе» при загрузке в ACE) — бытовой чат; «на столе» не путается с heated bed принтера.
- Ответ на clarify шуточной «моделью» (Kobra X Max 5G, giga blaster…) или продолжение бытовой темы — бот молчит, без «still uncertain» на английском.
- Вопросы про сторонний пластик (Bambu Lab) и PETG HF — короткий ответ из справочника, не оглавление `filament-and-resin` со score 100%.
- Реплики «одному богу известно какой зазор у втулки/пластика» — бытовой чат, не `filament-guide`; «какой» в зазоре не путается с выбором марки филамента.
- Вопросы про цену на AliExpress и «комбо с какой аськой» — бытовой чат/маркетплейс, не `ace-pro-filament-replacement-guide`.
- Короткий торг про ACE («10₽ вторая аська? Дорого?») — тот же фильтр, не гайд по замене катушки.
- Объявление с барахолки («ссылка на avito.ru · в тольятти появились аськи по адекватной цене») — `marketplace_promo`, не `ace-pro-filament-replacement-guide`; хосты Avito/Youla/Drom/Farpost ловятся наравне с AliExpress/Ozon.
- Новостной пресс-релиз об анонсе продукта («Creality представила новую сушилку SpacePi X4S…») — `_is_product_news_announcement`, не `filament-and-resin`; «бренд представил/анонсировал новинку» без help-intent игнорируется.
- Риторическая претензия к производителю («Иначе нахрена они кастрировали новые модели на поддержку первой аськи? Ведь технически это одинаковые устройства») — `_is_sarcastic_thread_banter`, не `ace-pro-filament-replacement-guide`; вульгарная риторика (нахрена/на кой) + претензия о моделях/совместимости — спор, не вопрос.
- «Продать? … ТПУ из аськи не сможет, сушить есть где» — бытовой чат о продаже ACE, не `ace-pro-notes`; «сушить есть где» не путается с вопросом про сушилку.
- Истории про P2S/Bambu, eSUN/«ориг» пластик и подтёки (техподдержка, брак) — бытовой чат, без clarify и без `extra-material` Kobra.
- `GIT_AUTOPULL_ENABLED`: фоновый `git pull` (по умолчанию выкл.; обновление вручную — `/update`).

## Служебный чат (ошибки, перезапуски, зеркало)

По умолчанию в Telegram-канал **AnycubicWikiBotLog** с id **`-1003826125815`** (`app/default_ops_chat.py`) бот отправляет:

- **зеркало лога** (`OPS_LOG_MIRROR_ENABLED`, по умолчанию вкл. если задан `OPS_NOTIFY_CHAT_ID`) — не полный дамп консоли, а отфильтрованные события:
  - **ответы бота** (`bot_reply`): вопрос пользователя, текст ответа, ссылки на сообщения в чате, score/url/код ошибки, **источник запроса** (`🎯 авто-вопрос` / `📣 упоминание` / `↩️ ответ боту` / `👤 личка`) и **модель принтера** (`🖨 slug`), определённая эвристикой — чтобы при разборе ложных ответов было видно, ответил ли бот сам или его позвали;
  - **негативные реакции** (`bot_reaction`): если на сообщение бота поставили 💩/👎 (по умолчанию только админ/разработчик чата — `REACTION_LOG_ADMIN_ONLY=1`), в зеркало приходит карточка с эмодзи, ссылками на вопрос и ответ бота, типом ответа и текстами — чтобы потом скинуть её на разбор и поправить эвристику. Набор «плохих» эмодзи: `NEGATIVE_REACTION_EMOJIS` (через пробел/запятую, дефолт `💩 👎`). Важно: чтобы Telegram присылал реакции в группах, бот должен быть **администратором** чата;
  - **старт** — компактная строка `startup_ready` (username, число страниц вики, QA, коды);
  - **индексация** — прогресс, завершение, автоподстройка интервала;
  - **git** — `/update`, autopull, перезапуск;
  - **не зеркалятся**: каждое входящее (`seen`, `Входящее сообщение`), `skip … reason=`, уточнения `clarify`, шум apscheduler/httpx.
- **необработанные исключения** в обработчиках (`on_error` через `ops_notify`);
- **перезапуск** перед остановкой polling после `/update` или фонового git autopull;
- **сбой `/update`** и исключение в фоновом `git autopull`.

Параметры зеркала: `OPS_LOG_MIRROR_ENABLED=0` — выкл.; `OPS_LOG_MIRROR_LEVEL=DEBUG`; `OPS_LOG_MIRROR_INTERVAL_SECONDS=2` (интервал слива буфера).

Отключить или сменить чат: **`OPS_NOTIFY_CHAT_ID`** (`0` / `off` — выключить; иначе числовой `chat_id`).

## Веб-панель администратора

Встроенная веб-панель (без внешних зависимостей, `app/web_panel.py`) поднимается в фоновом потоке **внутри того же процесса**, что и бот, поэтому стартует и останавливается вместе с ним — отдельный сервис не нужен.

Включается при **`PANEL_ENABLED=true`** (или автоматически, если задан `PANEL_PASSWORD`). Возможности:

- **Дашборд** — статус, размер индекса вики, счётчики записей, ключевые параметры конфигурации;
- **Ручные ответы** — просмотр / добавление / редактирование / удаление `data/manual_qa.json` (как `/qaadd`, `/qalist`, `/qadel`, но в браузере); правки сразу применяются к работающему боту (без перезапуска) и, при `MANUAL_QA_GIT_PUSH=1`, коммитятся и пушатся;
- **Фиксы ссылок** — просмотр / добавление / удаление переопределений (`fixes.json`, как `/fix`);
- **Логи** — хвост `logs/bot.log` с фильтром по подстроке (удобно ловить ложные срабатывания);
- **Обновление** — в шапке кнопки «Проверить обновления» (git fetch + сравнение с GitHub) и «Обновить» (git pull + перезапуск), как команда `/update`, но из браузера;
- **Настройки** — редактирование `.env` прямо из браузера (всё, кроме токена бота): пороги, антиспам, индексация, уточнения, список чатов, параметры панели и т.д. Кнопка «Сохранить» пишет в `.env`; «Сохранить и перезапустить» — ещё и перезапускает бота, чтобы применить (часть параметров — порт/адрес панели, индексация, лимиты — вступает в силу только после перезапуска). Пароль запасного входа меняется там же (пустое поле = не трогать).

### Вход

1. **Через бота — без домена** (рекомендуется). На странице входа кнопка «Войти через Telegram-бота»: открывается `t.me/<бот>?start=<код>`, вы жмёте **Start**, бот через `get_chat_member` проверяет, что вы **админ группы** `PANEL_ADMIN_CHAT_ID` (по умолчанию основная группа), и панель пускает вас. Работает на «голом» IP:порт, домен и HTTPS не нужны. Код одноразовый, живёт 5 минут и привязан к браузеру, где начат вход.
2. **По логину/паролю** — запасной способ, активен только если задан `PANEL_PASSWORD` (членство в группе не проверяет).
3. **Login Widget** (опционально, нужен домен) — endpoint `/tg-auth` поддерживает классическую кнопку «Log in with Telegram». Чтобы ей пользоваться, в [@BotFather](https://t.me/BotFather) → `/setdomain` укажите домен панели; на «голом» IP виджет Telegram не работает, поэтому по умолчанию в интерфейсе показан вход через бота.

Настройки (`.env`):

| Переменная | По умолчанию | Назначение |
|---|---|---|
| `PANEL_ENABLED` | `true`, если задан пароль | Включить панель (для TG-входа без пароля задайте явно `true`). |
| `PANEL_TG_LOGIN` | `true` | Вход через Telegram. |
| `PANEL_ADMIN_CHAT_ID` | `-1003881305021` | Группа, чьи админы могут входить. `0`/`off` — выключить TG-вход. |
| `PANEL_PASSWORD` | — | Пароль запасного входа. Пусто = вход только через Telegram. |
| `PANEL_USERNAME` | `admin` | Логин запасного входа. |
| `PANEL_HOST` | `0.0.0.0` | Адрес. `127.0.0.1` — только локально. |
| `PANEL_PORT` | `8080` | Порт. |
| `PANEL_SESSION_TTL_SECONDS` | `86400` | Срок жизни сессии. |

Безопасность: вход через Telegram проверяется HMAC-подписью и членством в админах группы; сессия — cookie (`HttpOnly`, `SameSite=Lax`); формы изменения — CSRF-токен; у входа троттлинг (8 неудач за 5 минут → пауза). Панель ходит по **HTTP** — при доступе снаружи закройте её reverse-proxy с HTTPS (nginx/Caddy); это заодно даёт домен для `/setdomain`.

## Логи на сервере (systemd + screen)

Если бот запущен через **`ensure-bot.sh`** в **screen**, вывод **`python -m app.bot`** попадает в **screen** и в файл **`logs/bot.log`**, а не в **`journalctl -u …`**. В журнале systemd обычно видны только строки скриптов при старте unit’а.

Смотреть лог бота:

```bash
sudo -u <пользователь_бота> tail -f /путь/к/KobraS1BotWiki/logs/bot.log
```

При **`LOG_DECISIONS=true`** в файле есть строки **`seen chat=`** и **`skip … reason=`** — по ним видно, почему бот не ответил на сообщение. В служебный Telegram-канал эти строки **не** попадают (только `bot_reply` и события из раздела выше).

## Git и команда `/update`

Команда **`/update`** выполняет `git fetch` / синхронизацию с remote от **пользователя процесса бота**. Если репозиторий или **`.git/objects`** когда‑то создавались от **root**, возможна ошибка *insufficient permission for adding an object*. Исправление: выдать владельцем каталог клона пользователя бота, например `chown -R user:user /path/to/KobraS1BotWiki`.

Опционально: **`GIT_AUTOPULL_ENABLED=1`** — периодическая проверка remote без ручного `/update` (интервал, ветка и перезапуск — в `app/config.py` и комментариях к переменным `GIT_AUTOPULL_*`).
