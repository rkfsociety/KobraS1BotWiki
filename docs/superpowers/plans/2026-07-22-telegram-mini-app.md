# Telegram Mini App для администраторов — план реализации

> **Для исполнителей:** перед началом реализации обязательно использовать `superpowers:subagent-driven-development` (рекомендуется) или `superpowers:executing-plans`. Выполнять задачи последовательно, с тестом и отдельным коммитом на логически завершённый блок.

**Цель:** добавить одно Telegram Mini App по адресу `/app`, которое определяет пользователя через Telegram, пускает администраторов единственной настроенной группы в админский интерфейс и позволяет разбирать вопросы без ответа.

**Архитектура:** Mini App будет отдельным мобильным HTML/CSS/JS-интерфейсом, обслуживаемым текущим Python HTTP-сервером. API приложения будет использовать существующие `bot_data`, очереди `missed_questions`, ручные ответы и статистику, поэтому бот, веб-панель и Mini App останутся синхронизированы. Бот и приложение работают только с одной группой из `PANEL_ADMIN_CHAT_ID`; пользовательский режим добавляется позже в том же приложении.

**Стек:** Python 3, существующий `http.server`, Telegram Bot API, vanilla JavaScript, HTML/CSS, pytest.

## Глобальные ограничения

- Не создавать новую git-ветку: вся работа выполняется в текущей ветке `master`.
- Не добавлять Telegram-токен, пароль панели, DuckDNS-токен или другие секреты в git.
- Telegram `initData` всегда проверяется на сервере через HMAC; данным из `initDataUnsafe` в браузере не доверять.
- Бот и Mini App обслуживают только одну группу, указанную в `PANEL_ADMIN_CHAT_ID`.
- Администратор MVP — только пользователь со статусом `creator` или `administrator` в этой группе.
- Если бот не может проверить группу или пользователь не администратор, API возвращает `401/403`, не раскрывая статистику.
- Вопрос без уверенного ответа сохраняется в существующую очередь `data/missed_questions.json` через общий код санитизации.
- Все изменения файлов репозитория сопровождаются тестами и git-коммитом; push и deploy выполняются только по прямому запросу.

### Задача 3: API и маршрутизация Mini App

**Файлы:**
- Создать: `app/web_miniapp.py`
- Изменить: `app/web_panel.py` в обработчиках `do_GET`, `do_POST` и общих функциях отправки ответа.
- Тест: `tests/test_web_miniapp.py`

**Интерфейсы:**
- `render_miniapp() -> bytes` — мобильная страница приложения.
- `GET /app` — HTML Mini App без старой cookie-сессии веб-панели.
- `POST /api/app/session` — принимает `initData`, возвращает `{user, role, capabilities}` после серверной проверки.
- `GET /api/app/dashboard` — возвращает только админские агрегаты после проверки сессии.
- `GET /api/app/missed` — возвращает очередь вопросов без ответа.
- Все JSON-ответы используют `application/json; charset=utf-8`, `Cache-Control: no-store` и безопасные коды ошибок.

## Отложенная вторая фаза: пользовательский режим

После подтверждения админского MVP добавить в том же `/app`:

- роль `user` для обычных участников;
- поиск по вики;
- форму вопроса боту;
- сохранение вопроса без ответа в общей очереди;
- ответ «Пока я не могу ответить…» при отсутствии уверенного результата;
- просмотр пользователем статуса своего вопроса без доступа к админским данным.

## Самопроверка плана

- Проверены границы: админский MVP и пользовательская фаза разделены, но используют один URL, общий API и одну Telegram-группу.
- Все задачи имеют конкретные файлы, интерфейсы, тесты и команды проверки.
- Секреты, права Telegram и текущая серверная схема явно учтены.
- В плане нет новых git-веток и нет разрешения на push/deploy без отдельного запроса.
