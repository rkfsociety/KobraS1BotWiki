# Дизайн пользовательского чата Mini App

## Цель

Превратить пользовательский режим Telegram Mini App в персональный чат с ботом. Каждый пользователь должен видеть только свою историю сообщений, получать ответы из текущей базы ручных ответов и вики, а вопросы без ответа должны попадать в общую очередь разбора администраторов.

## Границы

В этой итерации реализуется чат для одной настроенной Telegram-группы. Обычные пользователи получают доступ после проверки членства в этой группе. Администраторы сохраняют доступ к панели и могут переключаться в пользовательский режим для тестирования. История и лимиты общие для приложения, но разделены по Telegram ID пользователя.

## Выбранный подход

Используется SQLite-файл в `data/`. Он не требует отдельного сервиса, переживает перезапуск бота и безопаснее JSON при параллельных HTTP-запросах. Запись выполняется через транзакции, включается WAL-режим, а соединения создаются с коротким временем ожидания блокировки.

## Структура базы

### `chat_messages`

| Поле | Назначение |
| --- | --- |
| `id` | Автоинкрементный идентификатор |
| `user_id` | Telegram ID владельца истории |
| `role` | `user` или `bot` |
| `text` | Текст сообщения |
| `source` | `manual`, `wiki`, `missing`, `system` или `error` |
| `created_at` | Unix-время создания |
| `reply_to_id` | Связь ответа бота с сообщением пользователя |

Индекс `(user_id, id)` обеспечивает загрузку истории конкретного пользователя в правильном порядке.

### `rate_limit_events`

| Поле | Назначение |
| --- | --- |
| `id` | Идентификатор события |
| `user_id` | Telegram ID пользователя |
| `created_at` | Unix-время запроса |

Индекс `(user_id, created_at)` используется для проверки короткого и часового окна. Старые события периодически удаляются.

## API Mini App

- `GET /api/app/chat/history` — последние сообщения текущего пользователя с пагинацией по `before_id`.
- `POST /api/app/chat/message` — принять вопрос, проверить лимит, сохранить его, выполнить поиск и сохранить ответ бота.

Все endpoint используют существующую Bearer-сессию Mini App. Telegram ID берётся только из серверной сессии, поэтому клиент не сможет запросить или записать историю другого пользователя.

Ответ `POST /api/app/chat/message` содержит сообщение пользователя, сообщение бота, источник ответа и актуальный остаток лимита. При неизвестном вопросе возвращается штатный текст ожидания, а вопрос передаётся в `missed_questions` через существующий санитайзер.

## Интерфейс

Пользовательский режим отображается как лента диалога:

- сообщения пользователя выравниваются вправо;
- сообщения бота — влево;
- история загружается при открытии режима;
- старые сообщения можно подгрузить кнопкой сверху;
- поле ввода закреплено снизу и не вызывает горизонтальный скроллбар;
- во время запроса кнопка блокируется, отображается состояние поиска;
- ошибки сети показываются в ленте и не удаляют уже загруженную историю.

Администраторский переключатель остаётся доступен только в сессии с ролью `admin`. Возврат в панель не создаёт новую авторизацию.

## Защита от спама

Ограничения проверяются на сервере до запуска поиска:

- не более одного сообщения в 3 секунды;
- не более 20 сообщений за 10 минут;
- длина вопроса от 2 до 2000 символов;
- одинаковый вопрос в течение 10 секунд не запускает повторный поиск;
- при превышении лимита сервер возвращает `429` и `retry_after` в секундах.

Лимиты считаются по Telegram ID пользователя и не доверяют данным из JavaScript. Для повторов сохраняется пользовательское сообщение, но ответ не дублируется: клиент получает ранее сохранённый результат.

## Ошибки и восстановление

- Ошибка поиска не должна ронять HTTP-поток или бот; в историю записывается сообщение `error` с понятным текстом.
- Ошибка записи истории возвращает ошибку сервера и не создаёт частичный ответ.
- Если база ещё не создана, приложение создаёт каталог и таблицы при старте первого запроса.
- Ограничение размера истории на пользователя: хранить последние 500 сообщений; старые записи удаляются фоново или при записи новой пары сообщений.
- Содержимое сообщений проходит существующий санитайзер перед попаданием в общую очередь. В историю сохраняется исходный текст пользователя, так как это персональные данные приложения и нужен для отображения контекста.

## Тестирование

Нужны тесты на:

- создание схемы и изоляцию историй двух пользователей;
- загрузку истории и порядок сообщений;
- успешный ответ из manual QA и из вики;
- сохранение неизвестного вопроса в очередь;
- повторный запрос без дублирования ответа;
- короткий и часовой rate limit с ответом `429`;
- запрет доступа без действующей Mini App-сессии;
- отображение элементов чат-интерфейса и отсутствие небезопасных данных в HTML.

## Не входит в эту итерацию

- потоковые ответы по WebSocket/SSE;
- загрузка файлов и изображений;
- удаление отдельных сообщений пользователем;
- общая история группы;
- отдельная PostgreSQL-инфраструктура.
